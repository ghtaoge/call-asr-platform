# 实时语音识别、TTS 与异步分析设计

日期：2026-07-17

## 1. 背景

当前系统支持上传双声道录音或提交语音 URL，并在后台完成转写、情绪识别、敏感词与风险扫描、质检和 DeepSeek 摘要。现有后端保留了一个实时 WebSocket 原型，但它会把每个音频块作为完整离线文件重新分析，无法稳定输出临时文本，也会重复执行预处理和分析。前端目前没有实时识别入口。

离线任务虽然已经把 DeepSeek 摘要移到转写结果之后执行，但情绪、风险和质检仍属于同一条本地流水线。用户必须等本地分析全部完成，才能看到最重要的通话内容。

本设计同时解决三个问题：

1. 增加浏览器麦克风实时识别，并为 Gooeto 实时音频流预留稳定接口。
2. 增加基于阿里开源 CosyVoice 的文本转语音和参考音频声音复刻。
3. 把通话转写设为最高优先级，转写完成立即展示；情绪、风险、敏感词、质检和摘要独立异步执行。

## 2. 目标

### 2.1 实时识别

- 第一阶段支持浏览器麦克风采集。
- 后端定义与输入来源无关的音频帧协议，为后续 Gooeto 音频流适配器预留边界。
- 实时显示可修订的临时文字，停顿后固化为带起止时间的正式语句。
- 浏览器单声道同时录到双方声音时，先聚类为“说话人 1/2”，再由用户映射为销售和客户。
- 热启动 CPU 环境下，临时文字在 1 秒内更新，停顿后的正式语句在 2 秒内确认。

### 2.2 TTS

- 后端使用 CosyVoice，不依赖浏览器系统音色。
- 用户上传参考音频进行声音复刻。
- 支持在独立 TTS 页面输入任意文本，也支持对单句通话文本快速合成。
- 合成结果支持播放和下载。
- 参考音频与生成文件使用临时任务存储，并按现有任务保留期限清理。

### 2.3 转写优先与异步分析

- 上传文件、语音 URL 和实时会话共用同一种“转写先可用”的结果语义。
- 转写完成后立即保存并开放通话内容接口。
- 情绪、敏感词与风险、质检、DeepSeek 摘要各自拥有独立状态。
- 各分析模块逻辑并行，完成一个就展示一个；单个模块失败不影响通话内容或其他模块。
- 实时会话中，正式句子产生后立即异步扫描敏感词与逐句风险；完整情绪曲线、质检和摘要在通话结束后执行。

## 3. 非目标

- 本期不实现 Gooeto 真实生产音频流接入，只交付协议和适配器接口。
- 本期不建立长期音色库，不提供跨任务复用的声音身份管理。
- 本期不让模型直接猜测“销售/客户”；只做说话人聚类和人工角色映射。
- 本期不引入 Redis、Celery 或独立微服务部署。
- 本期不在每次临时文本变化时运行情绪、摘要或完整质检。
- 本期不提供多人会议级别的任意人数说话人分离；首期只支持两个主要说话人。

## 4. 总体架构

系统保持 FastAPI 单体部署，但新增清晰的服务边界：

### 4.1 RealtimeGateway

负责 WebSocket 生命周期和传输协议，不直接调用离线分析服务。

职责包括：

- 创建、恢复和结束实时会话。
- 校验音频格式、采样率、帧序号和会话状态。
- 接收 JSON 控制消息和二进制音频帧。
- 维护短时断线恢复所需的确认序号。
- 将规范化后的 `AudioFrame` 交给流式转写服务。
- 把领域事件序列化为前端 WebSocket 消息。

### 4.2 AudioSourceAdapter

所有实时音频来源转换为统一结构：

```text
AudioFrame
  session_id
  sequence
  captured_at_ms
  codec
  sample_rate
  channels
  payload
```

首期实现 `BrowserMicrophoneAdapter`。浏览器使用 `AudioWorklet` 采集音频，发送 16kHz、单声道 PCM 二进制帧，避免 Base64 的体积和编码开销。

后续 `GooetoAudioAdapter` 只负责鉴权、解包、时间戳和格式转换。ASR、分句、存储和前端事件协议不依赖 Gooeto 私有格式。

浏览器每 20ms 发送一帧。二进制帧使用固定 16 字节大端头部：1 字节协议版本、1 字节标志、2 字节保留位、4 字节无符号序号、8 字节采集时间戳，后面紧跟 PCM 载荷。首期协议版本为 `1`。

### 4.3 StreamingTranscriptService

负责一个实时会话内的流式 ASR 状态：

- 维护模型上下文和滚动音频窗口。
- 产生可修订的临时文本。
- 结合 VAD 在停顿后固化正式语句。
- 为正式语句分配稳定 ID 和起止时间。
- 调用两人说话人聚类器，为正式语句分配 `speaker_1` 或 `speaker_2`。
- 将正式语句写入 `SessionRepository`。
- 发布逐句敏感词和风险扫描任务。

临时文本不进入持久化结果，也不触发情绪、摘要或完整质检。只有正式语句是后续模块的稳定输入。

### 4.4 AnalysisOrchestrator

替换当前“所有本地分析完成后才返回”的单流水线语义。它把工作拆为：

1. 转写任务。
2. 敏感词与风险任务。
3. 情绪任务。
4. 质检任务。
5. DeepSeek 摘要任务。

转写完成后立即保存 `segments`，将 `transcript_status` 设置为 `completed`，并允许前端读取结果。后续四类任务分别调度、分别保存状态和产物。

“并行”表示状态和失败相互独立，不要求所有 CPU 模型同时抢占计算资源。调度器使用有界队列和资源优先级：

1. 实时 ASR。
2. 离线 ASR。
3. 实时逐句敏感词与风险。
4. 会后风险、质检和 DeepSeek 网络请求。
5. 情绪模型。
6. TTS。

DeepSeek 属于网络 I/O，可以与本地模型任务同时进行。CPU 环境存在实时会话时，新的 TTS 和非必要模型任务进入等待队列。

### 4.5 TtsService

TTS 与 ASR 使用独立任务和存储边界：

- 校验参考音频、授权确认、文本长度和任务配额。
- 延迟加载 CosyVoice 模型并串行化重型推理。
- 根据参考音频生成仅限当前保留期使用的临时 `voice_id`。
- 生成标准可播放音频文件。
- 提供状态、Range 播放和下载接口。
- 复用任务存储的过期清理机制。

## 5. 实时会话协议

### 5.1 客户端控制消息

开始会话：

```json
{
  "type": "start_session",
  "codec": "pcm_s16le",
  "sample_rate": 16000,
  "channels": 1,
  "source": "browser_microphone"
}
```

角色映射：

```json
{
  "type": "map_speakers",
  "mapping": {
    "speaker_1": "sales",
    "speaker_2": "customer"
  }
}
```

结束会话：

```json
{
  "type": "end_session"
}
```

暂停和恢复只控制浏览器采集，不销毁服务端 ASR 上下文：

```json
{ "type": "pause_session" }
{ "type": "resume_session" }
```

音频使用二进制帧发送。帧头包含协议版本、序号和采集时间；载荷是 PCM 数据。服务端定期返回最后确认序号。

### 5.2 服务端事件

- `session_started`：会话已就绪。
- `audio_ack`：已接收的最后连续音频序号。
- `partial_transcript`：当前临时文本及修订号。
- `final_transcript`：稳定的正式语句。
- `speaker_mapping_updated`：历史正式语句角色映射已更新。
- `risk_update`：单句敏感词和风险扫描完成。
- `session_ended`：音频输入结束，正式语句已全部固化。
- `analysis_status`：某个异步模块状态变化。
- `analysis_result`：某个异步模块产物可用。
- `error`：可恢复或不可恢复错误。

`partial_transcript` 使用单调递增的 `revision`。前端只用更新版本更高的事件替换当前临时行。`final_transcript` 到达后移除临时行并插入正式语句，避免重复内容。

### 5.3 断线恢复

- 浏览器保留最近 10 秒尚未确认的音频帧。
- 断线 30 秒内使用同一会话 ID 和服务端确认序号恢复连接；超过 30 秒后结束该会话并保留已固化文本。
- 客户端只补发确认序号之后的帧。
- 服务端按 `(session_id, sequence)` 去重。
- 超出短时缓存窗口时，保留已确认文本并提示部分音频可能缺失，不清空整个会话。

## 6. 任务与结果状态

任务状态增加独立模块字段：

```text
transcript_status: pending | running | completed | failed
emotion_status: pending | running | completed | failed
risk_status: pending | running | completed | failed
quality_status: pending | running | completed | failed
summary_status: pending | running | completed | failed
```

整体任务状态只用于表示任务是否仍有后台工作，不再作为读取通话内容的唯一门槛。

结果接口规则：

- `transcript_status` 未完成时，结果接口返回“转写尚未就绪”。
- `transcript_status=completed` 后，结果接口立即返回 `segments`。
- 尚未完成的分析产物为 `null` 或空集合，同时返回对应模块状态。
- 模块失败时返回稳定错误代码和中文提示，并允许仅重试该模块。
- 重试不会重新执行已经完成的转写。

实时会话在 `final_transcript` 到达时就能显示通话内容。会话结束后转入与离线任务相同的分析状态模型。

## 7. 页面设计

页面顶部增加模式切换：

```text
录音分析 | 实时识别 | 语音合成
```

### 7.1 实时识别

主操作区包含：

- 开始、暂停和结束录音控制。
- 麦克风权限和 WebSocket 连接状态。
- 录音时长和输入音量。
- 说话人 1/2 到销售/客户的映射控件。
- 实时通话内容列表。

临时文字只占用列表底部的一行，使用弱化样式表示尚未确认。正式语句沿用现有角色、时间点和高亮结构。页面自动跟随最新语句，但用户主动向上滚动后暂停自动跟随，并提供返回最新位置的按钮。

通话结束后，通话内容保持在主区域。敏感词逐句结果继续补充；情绪、风险汇总、质检和摘要区域分别展示“等待中”“分析中”“已完成”“失败”状态。

### 7.2 录音分析

上传文件或提交 URL 后，页面继续显示阶段进度。`transcript_status=completed` 时立即打开工作区并显示通话内容，不等待其他模块。

通话内容保持主视觉。情绪曲线、风险、质检和摘要显示独立加载状态。某一模块失败时仅在对应区域提供重试，不覆盖通话内容。

### 7.3 语音合成

独立 TTS 页面包含：

- 待合成文本输入。
- 参考音频上传。
- 声音使用授权确认。
- 合成操作和排队状态。
- 合成结果播放器。
- 下载按钮。

每句通话文本旁增加扬声器图标。用户已经上传有效参考音频时，可用当前临时音色快速合成该句；否则切换到 TTS 页面并保留待合成文本。

## 8. TTS 接口

### 8.1 创建临时音色

```text
POST /api/tts/voices/clone
Content-Type: multipart/form-data
```

输入包括参考音频和授权确认。返回临时 `voice_id`、过期时间和校验结果。

参考音频接受 WAV、MP3、M4A 和 AAC，时长为 3 至 30 秒，文件不超过 20 MB。服务端统一转换为 CosyVoice 所需格式，并在创建临时音色前确认检测到有效人声。

### 8.2 创建合成任务

```text
POST /api/tts/jobs
```

输入包括文本和 `voice_id`。返回 `202 Accepted`、任务 ID 和排队状态。

单个任务最多接受 2,000 个中文字符；更长内容由前端明确提示用户拆分，不在后端静默截断。

### 8.3 查询和播放

```text
GET /api/tts/jobs/{job_id}
GET /api/tts/jobs/{job_id}/audio
```

音频接口支持 HTTP Range 和下载文件名。任务状态包括 `queued`、`running`、`completed`、`failed` 和 `expired`。

## 9. 说话人聚类与角色映射

浏览器麦克风是单声道，不能像 Gooeto 双声道录音一样直接按声道绑定角色。

- 流式服务对正式语句使用滚动声纹聚类，限制为两个主要说话人。
- 聚类标签是 `speaker_1` 和 `speaker_2`，不包含业务含义。
- 用户映射为销售/客户后，服务端持久化映射并更新历史正式语句。
- 后续敏感词、风险、情绪、质检和摘要使用业务角色。
- 映射变更后，依赖角色的已完成分析产物标记为过期并重新调度；转写文本不重新识别。
- 无法稳定区分时显示“未区分”，不强行猜测业务角色。

未来 Gooeto 适配器收到双声道音频时不运行声纹聚类，而是沿用 Gooeto 固定声道协议直接绑定销售和客户。只有单声道来源进入两人聚类流程。

## 10. 安全与隐私

- 使用参考声音前必须确认拥有合法授权。
- 参考音频按第 8.1 节限制格式、20 MB 文件大小、3 至 30 秒时长和人声有效性。
- 音频路径、原始音频内容和合成音频内容不得写入日志。
- 任务目录使用不可预测 ID。本期沿用现有单用户本地部署边界；多用户账号鉴权不在本期范围内。
- 参考音频、临时音色数据和生成文件沿用当前 7 天任务保留期，过期后自动删除。
- 下载响应保留“AI 合成”来源元数据；页面明确标记复刻音色。
- WebSocket 只接受 20ms PCM 帧，单帧载荷不超过 4 KB，服务端未处理缓冲不超过 10 秒，单次会话最长 2 小时，防止内存无界增长。
- DeepSeek API Key、模型凭据和客户音频继续只保存在本地配置或任务数据目录，不进入 Git。

## 11. 错误处理

### 11.1 实时识别

- 麦克风权限拒绝：提示用户在浏览器中开启权限，不创建空会话。
- 不支持的浏览器或采样格式：显示兼容性提示并阻止开始。
- 网络短暂中断：自动重连并补发未确认帧。
- 模型繁忙：保持连接，显示排队状态，不静默丢弃音频。
- ASR 不可恢复失败：保留已经固化的语句，结束当前输入并允许重新开始。
- 说话人聚类失败：保留文本，角色显示“未区分”。

### 11.2 异步分析

- 每个模块使用独立错误码、错误提示和重试入口。
- 分析失败不修改 `segments`，也不把整体页面切换为失败页。
- DeepSeek 超时不影响本地分析结果。
- 角色映射变化导致的旧结果使用 `stale` 内部状态，重新分析完成前页面显示“正在按新角色更新”。

### 11.3 TTS

- 参考音频过短、过长、无人声或格式不支持时拒绝创建临时音色。
- 文本为空或超过 2,000 个中文字符时不创建任务。
- CosyVoice 加载或推理失败时保留任务错误，允许重新提交。
- 临时音色过期时要求重新上传参考音频，不尝试读取已清理文件。

## 12. 性能策略

- 实时 ASR 使用常驻模型和会话级缓存，禁止按音频块重新构造完整离线服务。
- WebSocket 发送二进制 PCM，避免 Base64 编解码和 33% 体积膨胀。
- 临时文本只更新当前行，正式语句采用稳定 ID，减少 Vue 列表重复渲染。
- 敏感词扫描属于轻量 CPU 工作，可在正式语句到达后立即执行。
- 情绪模型按正式语句批量执行，并与转写模型使用独立有界执行器。
- TTS 使用单独队列；实时 ASR 活跃时不启动新的重型 TTS 推理。
- 模型启动状态进入健康检查，页面能够区分“模型加载中”和“任务分析中”。

## 13. 测试策略

### 13.1 单元测试

- 音频帧序号校验、确认和去重。
- 临时文本修订号只增不减。
- VAD 固化后只生成一条正式语句。
- 说话人聚类映射和历史语句更新。
- 各模块状态转换和独立重试。
- TTS 输入校验、过期和清理。

### 13.2 集成测试

- WebSocket 持续发送音频，先收到临时文字，再收到正式语句。
- 乱序、重复和断线重连帧不会生成重复文本。
- 使用阻塞的情绪、风险、质检和摘要替身，证明转写结果可以先读取。
- 分别让每个分析模块失败，验证其他模块继续完成。
- 角色映射后重新调度依赖角色的分析，不重新执行 ASR。
- CosyVoice 适配器测试创建音色、生成音频、状态查询、Range 播放、下载和过期清理。

### 13.3 前端测试

- 麦克风权限拒绝和授权后的状态变化。
- 开始、暂停、恢复和结束实时会话。
- 临时行被正式语句替换且不重复。
- 用户向上滚动时暂停自动跟随，返回最新按钮恢复跟随。
- 转写先展示，其他面板独立加载、完成和失败。
- 说话人映射更新已有文本角色。
- TTS 上传、授权、排队、播放、下载和过期提示。

### 13.4 验收指标

- 热启动 CPU 环境下，临时文字正常在 1 秒内更新。
- 停顿后正式语句正常在 2 秒内确认。
- 离线 ASR 完成后，不等待情绪或 DeepSeek 即可看到通话内容。
- 任一分析模块失败时，通话内容保持可读可播放。
- 实时 ASR 活跃时提交 TTS，不导致实时文本明显停顿。
- 现有后端测试、前端测试和生产构建继续通过。

## 14. 实施顺序

1. 拆分任务状态和部分结果接口，让离线转写先展示。
2. 提取正式语句持久化和独立分析编排器。
3. 重写实时 WebSocket 协议和浏览器麦克风采集。
4. 增加流式 ASR、临时/正式文本和断线恢复。
5. 增加两人说话人聚类和人工角色映射。
6. 增加实时逐句敏感词与会后并行分析。
7. 增加 CosyVoice 临时音色、TTS 任务、播放和下载。
8. 完成性能回归、浏览器端到端测试和文档更新。

该顺序先交付用户最关心的“通话内容优先展示”，再增加实时链路和 TTS，降低一次性改动风险。
